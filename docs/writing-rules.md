# 工作手册写作规则

## 信息落点

| 信息 | 唯一落点 |
|---|---|
| 用户确认的长期产品语义 | `projectneed.md` |
| 决策、理由、后果和勘误 | `decisions/` |
| 当前实现、调用链、状态地图和验收设计 | `design/` |
| 已接受且未完成的工作 | `NOW.md` |
| 当前任务范围与副作用 | 任务合同或 handoff |
| 实现事实 | 代码、配置、schema、测试和 Git |
| 构建与使用入口 | `README.md` |

## 写作边界

- 需求使用稳定 ID；引用 ID，不在多处转述正文。
- 代码只能证明当前实现，不能证明产品本来应该如此。
- 推断和未知只写在设计调查、任务合同或问题列表，不写成 requirement。
- proposed、accepted、current 和 historical 必须明确标记。
- 数据语义必须说明怎样增加、更新、逻辑失效、物理删除和恢复，不能只列路径。
- 错误决策不删除；追加 amended 或 superseded 记录。
- `NOW.md` 不记录完成历史；落地即剔除。
- 新增或改名文档时，同一提交更新 `INDEX.md` 和全部入站链接。
- 文档与实现冲突时报告冲突，不通过静默改文档让当前代码显得正确。
